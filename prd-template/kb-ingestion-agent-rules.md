# 知识库入库 Agent 检查规则

> 本文档用于知识库入库前的质量检查。目标不是评审 PRD 方案，而是确保入库内容忠于来源、结构可检索、知识库内部一致。

## 1. 适用范围

适用于将上线后的最终 PRD、已归档变更记录、已上线接口说明、已确认错误码表、上线验收标准、最终设计稿 / 上线截图等稳定交付物整理进入 knowledge-base 前的检查。

评审意见、讨论过程、临时聊天记录、未确认草稿、个人判断、AI 推断、未上线版本的中间方案，不作为事实依据；它们最多只能作为检查线索，用来提醒核对稳定交付物。

本规则关注三件事：

- 来源事实是否被忠实保留。
- 同一事实在知识库中是否一致。
- 内容是否挂在正确页面、状态、弹窗、拦截、H5 或组件下面，方便后续检索和复用。

本规则不用于替代 PRD 评审，不要求重新判断产品方案是否合理，也不要求补齐来源资料没有提供的实现细节。

## 2. Agent 组成与优先级

知识库入库检查使用 3 个 Agent：

1. **Fact Guard：事实保真检查**
2. **Consistency Guard：一致性检查**
3. **Knowledge Structure Guard：知识归属 / 层级检查**

优先级：

```text
Fact Guard > Consistency Guard > Knowledge Structure Guard
```

含义：

- 事实保真优先级最高。
- 任何结构优化、阅读优化、一致性修正，都不得覆盖来源中的明确事实。
- 如果来源事实明确但表达不够好，只能在保留事实完整性的前提下调整表达。
- 如果来源事实不明确，必须标记为 Gap / 待确认，不能补写成确定规则。

## 3. Fact Guard：事实保真检查

### 3.1 检查目标

Fact Guard 的唯一目标是事实保真：检查入库内容是否忠于上线后的稳定交付物。

Fact Guard 不负责让内容更短，也不负责重写成更“顺”的表达。如果“简洁”和“保真”冲突，必须优先保真。

一句话原则：

```text
任何明确的状态值、错误码、渠道限制、按钮名、字段名、时间、次数、阈值、跳转目标和前端文案，都必须按来源保留；如果来源不明确，必须标记为 Gap / 待确认，不得为了简洁、阅读性或结构一致而压缩、抽象或补写事实。
```

### 3.2 事实来源范围

知识库入库仅使用上线后的稳定交付物作为事实依据。

可作为事实依据：

1. 上线后的最终 PRD。
2. 已合并 / 已归档的变更记录。
3. 已上线接口文档。
4. 已确认错误码表。
5. 已上线验收标准 / 测试用例。
6. 最终设计稿 / 上线截图。
7. 已入库的关联知识文档。

不可作为事实依据：

1. 评审意见。
2. 讨论过程。
3. 临时聊天记录。
4. 未确认草稿。
5. 个人判断。
6. AI 推断。
7. 未上线版本的中间方案。

评审意见只能作为检查线索，不能直接入库为事实。如果评审意见与上线后的最终 PRD、已归档变更记录或已上线接口文档冲突，以稳定交付物为准；若稳定交付物之间存在冲突，应标记为来源冲突 / 待确认，不得自行裁决。

### 3.3 必查项

| 检查项 | 要求 |
|---|---|
| 明确状态值 | 必须保留原始状态值，不得抽象压缩 |
| 错误码 | 必须保留错误码、含义、前端文案、处理方式 |
| 渠道限制 | 必须保留来源渠道、入口、用户类型等限制条件 |
| 按钮 / 操作名称 | 来源明确时应保留，不要改成泛称 |
| 接口字段 | 字段名、枚举值、入参、出参必须原样保留 |
| 时间 / 次数 / 阈值 | 必须保留具体数值和单位 |
| 页面 / 弹窗 / H5 名称 | 来源明确时应保留原页面名或展示名 |
| 跳转目标 | 来源明确时必须保留目标页面；不明确时标待确认 |
| 版本 / 来源差异 | 必须标注冲突或待确认，不能自行合并 |
| 不确定内容 | 必须标记 Gap / 待确认 |

### 3.4 严禁压缩事实

AI 入库最常见风险是为了“简洁”压缩事实。Fact Guard 必须把事实压缩作为高风险问题处理。

禁止把明确枚举压缩成概括词。

错误示例：

```text
KYC 状态不可继续时，展示 Verification unavailable。
```

正确写法：

```text
后端返回 KYC 状态机为 Under review / Rejected / Approved，或用户在 waitlist 中且来源渠道是 APP 时，展示 Verification unavailable 状态。
```

禁止把具体错误码压缩成“错误”。

错误示例：

```text
DTC 返回错误时展示对应提示。
```

正确写法：

```text
DTC 返回 01009：Toast `Mobile number already exists.`
DTC 返回 01005：Toast `The email address is in use.`
DTC 返回 50013：Reverse Solicitation 缺失，阻止生成验证 URL。
```

禁止把具体阈值压缩成泛化表达。

错误示例：

```text
Face 多次失败后锁定。
```

正确写法：

```text
24 小时内 face fail 5 次：锁 20 分钟。
24 小时内 face fail 10 次：锁 24 小时。
24 小时内接口层连续发起 20 次：锁 20 分钟。
```

禁止把具体文件限制压缩成“文件过大 / 格式错误”。

正确写法必须保留：

```text
JPG / JPEG / PNG / PDF，单文件不超过 16MB。
```

禁止把渠道、入口、用户范围省略。

错误示例：

```text
用户在 waitlist 中时展示 Verification unavailable。
```

正确写法：

```text
用户在 waitlist 中，且来源渠道是 APP 时，展示 Verification unavailable。
```

### 3.5 不得把未知写成确定事实

只要来源没有明确页面、弹窗、toast、接口、文案、跳转目标，就不能写成确定事实。

错误示例：

```text
点击返回后回到 KYC Start Page。
```

如果来源没有明确，应写：

```text
返回目标源文档未明确。当前仅确认用户不能继续后续 KYC，具体返回 KYC Start 还是业务入口需产品 / UI 确认。
```

不得自行决定异常展示为 toast、弹窗、错误页或页面状态，除非稳定交付物明确说明。

### 3.6 可以补充关联事实，但必须满足条件

可以从关联事实文档补充内容，例如：

- 页面章节中提到错误码，可以补充错误码表中的文案。
- 页面章节中提到接口字段，可以补充接口文档中的字段名。
- 页面章节中提到 waitlist，可以补充 waitlist 数据落库字段。

补充必须满足：

1. 补充内容来自稳定交付物或已入库关联知识文档。
2. 补充内容和当前页面 / 状态 / 弹窗直接相关。
3. 补充后不能改变原始规则含义。
4. 如果多个来源冲突，不能自行融合，必须标冲突。

### 3.7 来源冲突处理规则

如果稳定交付物之间存在冲突，Fact Guard 不能自行裁决。

处理规则：

1. 保留冲突事实。
2. 标明冲突点。
3. 只写可以共同确认的结果。
4. 不自行裁定哪个来源更正确，除非有明确最新结论。

示例：

```text
源文档同时出现“waitlist 弹窗拦截”和“waitlist 调整为页面级拦截”的变更记录。
当前仅确认结果：用户不能继续 KYC，并可进入 Waitlist Page。
具体展示形态以最新 UI / 产品确认为准。
```

### 3.8 页面截图事实使用规则

页面截图可以作为事实来源，但只能证明“可见内容”，不能反推业务逻辑。

可以写：

```text
页面展示 Retry 和 Leave 按钮。
```

不能写：

```text
点击 Retry 一定重新提交 Face 结果。
```

除非最终 PRD、流程说明、验收标准或接口说明明确写了 Retry 的行为。

### 3.9 改写规则

知识库允许改写，但改写必须满足：

1. 不改变判断条件。
2. 不减少枚举值。
3. 不省略限制条件。
4. 不合并不同场景。
5. 不把多个结果写成一个泛化结果。
6. 不把来源未明确的内容写成确定事实。
7. 不为了阅读性删除来源里的例外、前提、阈值、状态值、错误码。

允许改写示例：

```text
来源：状态2：当后端返回kyc状态机为【Under review / Rejected / Approved】或用户在waitlist中（仅限来源渠道是APP），那么展示状态2内容。

允许改写：当后端返回 KYC 状态机为 Under review / Rejected / Approved，或用户在 waitlist 中且来源渠道是 APP 时，展示状态 2：Verification unavailable。
```

不允许改写：

```text
KYC 状态不可继续时，展示 Verification unavailable。
```

原因：压缩了状态枚举，并省略了 waitlist 来源渠道限制。

### 3.10 高风险词检查

Fact Guard 应扫描以下词。出现时不一定错，但必须检查是否压缩事实。

```text
或
对应
相关
失败处理
错误处理
按规则
按接口返回
根据状态
不符合条件
不可继续
异常状态
业务入口
来源页面
后续流程
适当提示
统一处理
```

| 高风险词 | 检查问题 |
|---|---|
| 或 | 是否本来是两个明确分支？ |
| 对应 | 是否缺少具体错误码 / 文案 / 页面？ |
| 失败处理 | 是否缺少失败类型和目标页？ |
| 按规则 | 规则在哪里？是否有引用？ |
| 不符合条件 | 条件是什么？ |
| 不可继续 | 哪些状态不可继续？ |
| 业务入口 | 具体哪个入口？来源是否明确？ |
| 统一处理 | 是否有统一处理文档引用？ |

### 3.11 阻断规则

以下问题必须阻断入库：

| 阻断问题 | 示例 |
|---|---|
| 明确枚举被压缩 | `Under review / Rejected / Approved` 被写成“不可继续” |
| 明确限制条件被省略 | `仅限 APP 来源` 被删掉 |
| 错误码被概括 | `01009 / 01005` 被写成“DTC 错误” |
| 数值阈值被泛化 | `16MB` 被写成“文件过大” |
| 来源没有的规则被补写 | 未说明自动跳转，却写成自动跳转 |
| 冲突来源被自行裁决 | 弹窗 / 页面级拦截冲突被写成单一确定形态 |
| Gap 被写成确定事实 | 待确认返回目标被写成固定页面 |
| 截图内容被反推业务规则 | 看到按钮后补写按钮逻辑 |
| 评审意见被当作事实 | 讨论意见覆盖上线后最终 PRD |

### 3.12 非阻断但必须提示的问题

| 问题 | 处理 |
|---|---|
| 表达不够清楚，但事实未错 | 建议调整表达 |
| 内部引用缺失 | 建议补充来源章节或关联文档 |
| 原文很长但完整保留 | 不阻断，不允许为了简洁压缩关键事实 |
| 同一事实重复出现 | 提醒 Consistency Guard 检查一致性 |
| 事实归属不清 | 提醒 Knowledge Structure Guard 检查挂载位置 |

### 3.13 Fact Guard 输出格式

```md
# Fact Guard 检查结果

## 1. 检查结论

- 是否通过：是 / 否
- 是否存在阻断问题：是 / 否
- 阻断问题数量：x
- 需人工确认问题数量：x

## 2. 阻断问题

| 位置 | 问题类型 | 当前写法 | 来源事实 | 为什么阻断 | 修复建议 |
|---|---|---|---|---|---|

## 3. 需人工确认问题

| 位置 | 当前写法 | 不确定点 | 建议标注 |
|---|---|---|---|

## 4. 可优化但不阻断

| 位置 | 当前写法 | 建议 |
|---|---|---|
```

## 4. Consistency Guard：一致性检查

### 4.1 检查目标

Consistency Guard 的目标是检查知识库内部同一事实是否前后一致。

它只检查同一事实在不同地方有没有冲突、重复、口径漂移，不重新判断业务方案，也不自行裁决哪个来源更正确。

一句话原则：

```text
Consistency Guard 只检查知识库内部同一事实是否前后一致，不自行裁决冲突，不用统一表达覆盖来源事实。凡涉及来源事实冲突、版本差异、缺少依据的问题，必须回到 Fact Guard 或标记为来源冲突 / 待确认。
```

Consistency Guard 和 Fact Guard 的边界：

| Agent | 职责 |
|---|---|
| Fact Guard | 检查这条事实是否忠于上线后的稳定交付物 |
| Consistency Guard | 检查同一条事实在知识库内部是否前后一致 |

### 4.2 必查对象

#### 4.2.1 页面名称一致性

同一页面在全文中名称必须一致。

错误示例：

```text
KYC Start Page
Start Page
KYC Start 主页面
认证开始页
```

如果它们指的是同一个页面，应统一使用来源中的主名称，例如：

```text
KYC Start Page
```

检查规则：

- 页面名称不得随意翻译、简称、改写。
- 如果为了中文说明需要补充，可写成 `KYC Start Page（认证开始页）`，但主名称必须保持一致。

#### 4.2.2 页面层级一致性

同一个页面不能一处作为页面级章节，另一处作为别的页面下的普通状态。

错误示例：

```text
### 4.8 Face Loading Page

另一个地方写：
Face Guide Page 下的 Face Loading 状态
```

检查规则：

- 页面级内容在全文中保持同层。
- 弹窗 / 拦截 / H5 / 组件必须稳定挂在所属页面下。
- 不得在不同位置改变其层级身份。

#### 4.2.3 状态值一致性

同一状态值必须保持原始写法。

错误示例：

```text
Under review
Under Review
under review
审核中
reviewing
```

如果来源是：

```text
Under review / Rejected / Approved
```

知识库中应统一保留这个写法。

检查规则：

- 状态值、枚举值大小写和空格应尽量与来源一致。
- 不得一处使用原文，一处使用翻译，一处使用泛称。

#### 4.2.4 错误码一致性

同一个错误码在页面章节、接口章节、错误码表中必须一致。

示例：

```text
01009
```

如果错误码表写：

```text
01009：Mobile number already exists.
```

页面章节不能写成：

```text
01009：手机号已绑定
```

除非明确标注这是中文解释，而不是前端文案。

推荐写法：

```text
DTC 返回 01009：Toast `Mobile number already exists.`
含义：手机号已存在。
```

检查规则：

- 错误码、英文前端文案、中文解释要分开。
- 不得把中文解释替代成前端文案。
- 不得在不同章节给同一错误码写不同处理方式。

#### 4.2.5 Toast / 前端文案一致性

来源明确的前端文案必须全文一致。

错误示例：

```text
Something went wrong. Please try again later
Something went wrong, please try again later.
Something went wrong. Try again later.
```

检查规则：

- 来源中明确的 toast、按钮文案、错误提示，应保持完全一致。
- 大小写、标点、缩写不得随意改。
- 如果文案来源不明确，应标：`具体文案以最终 UI / 文案表为准。`

#### 4.2.6 跳转目标一致性

同一个动作在不同章节中的跳转目标必须一致。

错误示例：

```text
KYC Start Page：
点击 Join waitlist 进入 Waitlist Page。

Waitlist Page：
来源是 waitlist 弹窗，返回 KYC Start。

流程图：
Join waitlist 返回业务入口。
```

检查规则：

- 同一个按钮 / 操作 / 状态结果，只能有一个确定跳转目标。
- 如果来源资料没有明确，所有相关位置都应统一标“源文档未明确 / 待确认”。
- 不能一处写待确认，另一处写确定页面。

#### 4.2.7 条件和结果一致性

同一个规则的条件和结果要完整一致，不能一处少条件，一处多条件。

错误示例：

```text
位置 A：
用户在 waitlist 中时展示 Verification unavailable。

位置 B：
用户在 waitlist 中，且来源渠道是 APP 时展示 Verification unavailable。
```

正确要求：所有位置都必须保留“来源渠道是 APP”。

检查规则：

- 如果某条规则有多个条件，所有引用位置都必须保留完整条件。
- 不得只保留结果，不保留触发条件。

#### 4.2.8 Gap 一致性

同一个 Gap 不能在一个地方写待确认，另一个地方写成确定事实。

错误示例：

```text
4.3：
返回目标源文档未明确。

4.5：
点击返回后回到 KYC Start Page。
```

正确做法：所有相关位置都标：

```text
返回目标源文档未明确，需产品 / UI 确认。
```

检查规则：

- 凡是已标 Gap / 待确认的事实，全文不能再写成确定规则。
- 如果后续有稳定交付物确认，必须同步更新所有引用位置。

#### 4.2.9 Source / 关联章节一致性

同一条知识引用的来源或关联章节应一致。

错误示例：

```text
4.3 Waitlist 拦截：见 4.5 Waitlist Page
另一个地方：见 4.6 Waitlist Page
```

检查规则：

- 章节编号、标题、锚点必须能对应到实际章节。
- 章节重排后必须同步更新所有内部引用。

#### 4.2.10 图片 / 页面说明一致性

同一截图不能在多个地方代表不同页面，除非明确说明是复用页面或通用状态。

错误示例：

```text
image6.jpeg 在 4.2 表示 KYC Loading Page
image6.jpeg 在 4.8 表示 Face Loading Page
但未说明它是通用 loading 样式
```

正确做法：

```text
该图为通用 loading 样式，分别用于 KYC Loading 和 Face Loading。
```

检查规则：

- 页面截图和页面说明要匹配。
- 同图复用必须解释复用关系。
- 不得让读者误以为两个不同页面共用同一个业务状态。

### 4.3 不允许的修复方式

Consistency Guard 不能这样做：

- 为了统一，把来源中的明确状态值改成泛称。
- 为了统一，把一个冲突来源直接删掉。
- 为了统一，把 Gap 写成确定事实。
- 为了统一，把不同场景强行合并。
- 为了统一，把前端文案翻译成中文。

错误修复示例：

```text
Under review / Rejected / Approved
统一改成：不可继续状态
```

这是事实压缩，违反 Fact Guard。

正确修复示例：

```text
保留完整枚举，并同步到所有引用位置。
```

### 4.4 冲突处理规则

当发现不一致时，按三类处理。

#### 4.4.1 知识库表达不一致，但来源事实明确

处理方式：按来源事实修正所有位置。

示例：

```text
来源明确：仅限来源渠道是 APP。

位置 A 缺失 APP 条件。
位置 B 保留 APP 条件。

修复：位置 A 补回 APP 条件。
```

#### 4.4.2 来源资料本身存在冲突

处理方式：

1. 不能自行裁决。
2. 保留冲突说明。
3. 只写共同确认结果。
4. 标记来源冲突 / 待确认。

示例：

```text
来源 A：waitlist 弹窗拦截
来源 B：waitlist 调整为页面级拦截

可确认结果：用户不能继续 KYC，可进入 Waitlist Page。
展示形态：来源冲突，需以最新 UI / 产品确认。
```

#### 4.4.3 知识库里一处是 Gap，一处是确定事实

处理方式：

- 如果没有新的稳定交付物确认，则把确定事实改回 Gap。
- 如果已有稳定交付物确认，则补充来源，并同步更新所有 Gap。

### 4.5 高风险一致性检查清单

Consistency Guard 应重点扫描这些内容：

```text
Page / 页面名
Status / 状态值
Error code / 错误码
Toast / 前端文案
Button / 按钮名
H5 名称
Waitlist
Declaration
reverseSolicitation
Phone / email duplicate
Face fail count
POA country mismatch
Loading timeout
Retry / Leave / Back / Close
Gap / 待确认 / 源文档未明确
```

出现这些内容时，应检查：

- 它在其他章节是否也出现？
- 写法是否一致？
- 条件是否完整？
- 结果是否一致？
- 是否一处待确认、一处确定？
- 是否引用到正确章节？

### 4.6 阻断规则

以下问题应阻断入库：

| 阻断问题 | 示例 |
|---|---|
| 同一事实前后冲突 | Waitlist 一处写弹窗，一处写页面，且未标来源冲突 |
| 同一错误码多种文案 | `01009` 一处写手机号已存在，一处写邮箱已存在 |
| 同一状态多种处理结果 | `Rejected` 一处进入 unavailable，一处进入 KYC Start |
| Gap 被其他章节写成确定事实 | 返回目标一处待确认，一处写回 KYC Start |
| 章节引用失效 | 说明见 4.10，但 4.10 已变成 Face Failed Page |
| 页面归属冲突 | Address Upload 的 waitlist 拦截被挂到 KYC Start 下 |
| 前端文案不一致 | 同一 toast 多种标点 / 大小写 / 文案 |
| 条件缺失导致规则变宽 | 一处省略“仅限 APP 来源” |

### 4.7 非阻断但必须提示的问题

| 问题 | 处理 |
|---|---|
| 页面名中英文混用但不影响理解 | 建议统一主名称 |
| 同一事实重复出现很多次 | 建议保留必要重复，避免改成抽象引用导致事实压缩 |
| 同图复用未说明 | 建议补充复用说明 |
| 章节编号引用不够友好 | 建议补链接或标题 |
| 同一规则引用多个来源 | 建议标明主来源和补充来源 |

### 4.8 Consistency Guard 输出格式

```md
# Consistency Guard 检查结果

## 1. 检查结论

- 是否通过：是 / 否
- 是否存在阻断问题：是 / 否
- 阻断问题数量：x
- 需同步修复问题数量：x

## 2. 阻断问题

| 事实点 | 冲突类型 | 位置 A | 写法 A | 位置 B | 写法 B | 建议处理 |
|---|---|---|---|---|---|---|

## 3. 需同步修复问题

| 事实点 | 涉及位置 | 当前不一致 | 统一规则 | 修复建议 |
|---|---|---|---|---|

## 4. 非阻断建议

| 位置 | 问题 | 建议 |
|---|---|---|
```


## 5. Knowledge Structure Guard：知识归属 / 层级检查

### 5.1 检查目标

检查内容是否挂在正确的页面、状态、弹窗、拦截、H5 或组件下面。

Knowledge Structure Guard 关注：**读者能不能知道这条知识属于哪个上级场景，并能不能快速检索到。**

### 5.2 页面章节分层原则

页面级别内容保持同一层级。

正确示例：

```md
### 4.2 KYC Loading Page
### 4.3 KYC Start Page
### 4.4 Select Residence Country Page
### 4.5 Waitlist Page
```

不要为页面本身再创建“主页面”子章节。

错误示例：

```md
### 4.3 KYC Start Page

#### 4.3.1 主页面
```

正确示例：

```md
### 4.3 KYC Start Page

页面截图 + 页面说明

#### 4.3.1 弹窗：Declaration of Reverse Solicitation
#### 4.3.2 拦截：Waitlist
```

### 5.3 子章节拆分规则

页面内的区域、控件、按钮、toast、简单弹窗、简单状态，优先写在当前页面说明里。

只有当某个内容在当前页面说明中写不清楚，或会导致说明过长、阅读困难，才拆成当前页面下的子章节。

| 情况 | 处理 |
|---|---|
| 一两句话能讲清楚 | 放在当前页面说明里 |
| Toast / 简单提示 | 放在当前页面说明里 |
| 按钮置灰 / 小状态变化 | 放在当前页面说明里 |
| 有独立截图 | 倾向拆下级章节 |
| 有多个按钮 / 多种操作结果 | 倾向拆下级章节 |
| 有保存规则 / 接口参数影响 | 倾向拆下级章节 |
| 写在右侧说明里会显得很长很乱 | 拆下级章节 |
| QA 需要单独测 | 拆下级章节 |

子章节标题使用类型前缀，避免被误认为页面级章节：

```md
#### 弹窗：xxx
#### 拦截：xxx
#### H5：xxx
#### 组件：xxx
#### 状态：xxx
```

### 5.4 状态内层级规则

状态、按钮、返回、异常不能突然变成同级说明。应按层级归属表达。

推荐结构：

```text
状态 / 区域
  - 进入前提
  - 展示内容
  - 用户操作
    - 操作 A：结果 A
    - 操作 B：结果 B
  - 异常 / 边界
```

错误示例：

```text
Loading Failed Page
- 点击 Retry：重新查询 Face 结果
- 点击 Leave：返回业务入口
- 查询失败：进入 Face Failed
```

正确示例：

```text
Loading Failed 状态
- 进入前提：Face Loading 超过 30 秒仍未收到结果。
- 展示内容：展示 Loading Failed 状态，并提供 Retry / Leave 操作。
- 用户操作：
  - 点击 Retry：重新提交或重新查询 Face 结果。
  - 点击 Leave：返回业务入口，不继续等待。
- 重试后结果：
  - 查询返回 Face 失败：进入 Face Failed Page。
  - 查询返回网络 / 系统异常：源文档未明确，需按统一错误处理确认。
```

### 5.5 异常与边界放置规则

异常、边界、Gap 跟随它影响的页面区域 / 弹窗 / 拦截态，不单独汇总。

| 内容 | 放置位置 |
|---|---|
| 协议获取失败 | KYC Start Page 的协议区说明 |
| 协议保存失败 | KYC Start Page 的底部认证按钮说明 |
| Declaration 关闭不算完成 | 弹窗：Declaration |
| reverseSolicitation 缺失 / 50013 | 弹窗：Declaration |
| Waitlist 展示形态冲突 | 拦截：Waitlist |
| 手机号未绑定 | KYC Start Page 的入口边界 |

不要单独创建泛化章节：

```md
#### 异常与边界
```

除非该异常确实无法归属到任何页面、状态、弹窗、拦截或组件。

### 5.6 Knowledge Structure Guard 输出格式

```md
## Knowledge Structure Guard 检查结果

| 位置 | 问题 | 当前归属 | 建议归属 |
|---|---|---|---|
| 4.3.1 主页面 | 页面本身被重复拆成子章节 | KYC Start Page > 主页面 | 删除“主页面”子章节，内容直接放在 4.3 下 |
```

## 6. 入库检查执行顺序

推荐顺序：

```text
1. Fact Guard
2. Consistency Guard
3. Knowledge Structure Guard
```

执行说明：

1. 先确认事实没有被改错、压缩、补写。
2. 再确认同一事实在知识库内部没有冲突。
3. 最后调整内容归属和层级结构。

如果三个 Agent 的建议冲突，以 Fact Guard 为准。

## 7. 阻断规则

以下问题应阻断入库：

| 阻断类型 | 示例 |
|---|---|
| 明确事实被压缩 | Under review / Rejected / Approved 被写成“不可继续状态” |
| 来源没有的规则被写成确定事实 | 未明确返回目标，却写成“返回 KYC Start Page” |
| 错误码 / 状态值 / 渠道限制丢失 | APP 来源 waitlist 条件被省略 |
| 同一事实在知识库内冲突且未标注 | waitlist 同时写成弹窗和页面，但未解释冲突来源 |
| 内容归属错误导致无法检索 | Address Upload 的 waitlist 拦截被挂到 KYC Start Page 下 |

非阻断但需要修复的问题：

| 类型 | 示例 |
|---|---|
| 表达不够清楚 | 操作结果缺少触发前提 |
| 层级不够清晰 | 状态内按钮被写成同级说明 |
| 内部链接缺失 | 提到错误码但没有指向错误码章节 |

## 8. 推荐入库检查报告格式

```md
# 知识库入库检查报告

## 1. 检查结论

- 是否可入库：是 / 否
- 是否存在阻断问题：是 / 否
- 需修复后入库的问题数：x

## 2. Fact Guard 结果

| 位置 | 问题类型 | 当前写法 | 来源事实 | 建议修正 |
|---|---|---|---|---|

## 3. Consistency Guard 结果

| 事实点 | 冲突位置 A | 冲突位置 B | 冲突内容 | 建议处理 |
|---|---|---|---|---|

## 4. Knowledge Structure Guard 结果

| 位置 | 问题 | 当前归属 | 建议归属 |
|---|---|---|---|

## 5. 入库前必须修复项

- [ ] ...

## 6. 可入库但建议后续优化项

- [ ] ...
```
