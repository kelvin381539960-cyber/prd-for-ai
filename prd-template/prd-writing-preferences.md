# PRD Writing Preferences PRD 写作长期偏好

本文用于沉淀长期 PRD 写作偏好、经验教训和避免重复踩坑的规则。

每次进入 PRD 写作流程时，AI 必须先读取本文，并在 PRD 完成后根据本次沟通总结是否有新增偏好需要沉淀。

---

## 1. 核心写作偏好

- PRD 主要面向前端、后端、测试、设计和产品，优先写可开发、可测试、可验收、可落地的规则。
- 不写解释型废话；不能被开发实现或测试判断对错的内容，默认不写。
- 页面能从原型直观看到的内容，不重复写成规则。
- 章节按需保留，不为了模板完整而补内容。
- 简单页面型需求不强制写验收标准 / 测试场景；页面规则和校验规则足够清楚时可删除。
- PRD 写完后必须做一次落地评审，评审重点是是否能实际交给前端、后端、测试、设计和产品推进。

---

## 2. Canvas 草稿协作偏好

- Canvas 是草稿协作区，不是正式产物，也不是 harness 强校验对象。
- brief / PRD / 评审默认先在 Canvas 中输出，方便用户快速反复修改。
- 用户确认前，不要把 Canvas 草稿写入 Git。
- Canvas 不需要在 Git 中记录 Canvas ID、确认时间或授权字段。
- validator 只校验 Git 中的正式 PRD 文件，不校验 Canvas。

---

## 3. 文件最小化偏好

- Git 只沉淀确认后的最小必要产物。
- 默认只写正式 PRD 正文：`requirements/YYYY-MM/<module>/<feature>.md`。
- brief、research、review、模块 `_index.md` 均为按需文件，不强制创建。
- brief 主要用于确认方向；如果 PRD 已吸收 brief 结论，且不需要长期追溯，可以不写入 Git。
- research 只有在调研量大、来源复杂、方案比较重要或后续要复用时才写入 Git。
- review 默认在 Canvas / 聊天中完成；只有用户要求沉淀，或 PRD 已进入正式 review / approved 阶段时才写入 Git。
- 模块 `_index.md` 只在一个需求拆成多个 PRD，或模块内 PRD 关系复杂时维护。

---

## 4. 页面与交互偏好

- 每个新增或改造页面建议有低保真原型；如果暂时没有原型，必须写清关键页面结构，不要编造图片路径。
- 页面说明推荐结构：原型图 → 页面目的 → 用户体验要点 → 展示规则 → 交互与校验规则。
- 展示规则只写必要规则；页面上已经清楚可见的布局、按钮、普通文案不写。
- 输入页重点写校验规则、错误结果、按钮可用条件、成功后流转。
- 成功页重点写业务结果、数据变化、返回后展示和必要的成功后处理。

---

## 5. 页面章节分层与左图右说明偏好

- PRD 目录要先按页面级别分层；页面级别的内容保持同一层级。
- 页面章节本身就是该页面，不要再创建“主页面”子章节。例如不要写 `### KYC Start Page` 后再写 `#### 主页面`。
- 页面内的区域、控件、按钮、toast、简单弹窗、简单状态，优先写在当前页面说明里；如果几句话能讲清楚，不要拆下级章节。
- 如果页面内某个弹窗、拦截态、复杂组件或复杂状态写在当前页面说明里会太长、太乱，或有独立截图、多按钮、多结果、保存规则、接口参数影响、QA 需单独测试，则在当前页面下创建子章节。
- 下级章节必须让人一眼识别它不是页面级章节，推荐命名：`弹窗：xxx`、`拦截：xxx`、`组件：xxx`、`状态：xxx`。
- 复杂弹窗、拦截态等子章节仍归属当前页面，不作为新的页面级章节。
- 页面说明优先采用左图右说明：左侧放页面 / 弹窗 / 拦截截图，右侧放该展示单元对应的规则、触发、跳转、异常和边界。
- 大屏阅读体验优先：图片可以适当放大，页面图置顶，允许横向滚动；不要为了避免横向滚动把内容拆碎到多个位置。
- 右侧说明不要复述页面上显而易见的静态内容，例如“页面区域：按钮、标题、输入框”；应直接写该区域如何工作、触发后去哪、异常怎么处理。
- 当前页面直接发生的事写一级；点击后进入的新页面 / 弹窗 / 选择结果写下一级，避免把子流程和当前页面展示混成同级。
- 异常、边界、Gap 跟随它影响的页面区域 / 弹窗 / 拦截态，不单独汇总成“异常与边界”章节；除非确实无法归属到任何展示单元。
- 说明中提到其他页面、弹窗、拦截态、错误码或公共能力时，要给明确章节指引，例如“触发 waitlist 拦截；见 4.3.2 拦截：Waitlist。提交页细则见 4.5 Waitlist Page。”
- 写作粒度以“人能快速定位、开发能一次理解并实现、QA 能直接测试”为准；不要为了完整而完整。

---

## 6. 用户体验偏好

PRD 不只要能开发和测试，也要让用户体验顺畅。每次评审必须检查：

- 用户是否能清楚知道当前在哪一步、为什么要做这一步、下一步会发生什么。
- 关键操作前后是否有明确反馈，例如发送成功、验证失败、保存成功、处理中、失败可重试。
- 表单输入、错误提示、按钮状态、返回路径是否符合用户预期。
- 敏感操作是否有足够解释，但不过度打断用户。
- 失败、锁定、过期、不可用等场景是否给用户可理解的下一步，而不是只显示错误。
- 成功后是否让用户知道业务结果已经生效，以及后续会影响什么。
- 是否存在不必要的步骤、重复确认、绕路、信息过载或让用户困惑的页面。

---

## 7. 复用页面 / 公共能力偏好

- 复用页面只写流转去向，不写场景参数、展示字段和公共页面内部规则。
- 公共能力已有规则只引用，不重复定义。
- 不要让开发误以为复用页面需要被改造。
- 对 OTP、密码、BIO、Notification 等公共能力，业务 PRD 只写本需求的入口、成功流转、失败后保持的业务结果。

示例：

| 场景 | 后续流转 |
|---|---|
| 验证成功 | 进入下一业务页面 |
| 验证失败 / 锁定 / 过期 / 重发超限 | 按公共能力规则处理；业务结果不变 |

---

## 8. 字段、接口与数据偏好

- 内部字段、内部接口、技术实现细节默认不写进 PRD。
- 仅当涉及外部系统、第三方接口、跨模块同步、对外字段、新增数据口径或会影响产品验收的数据变更时，才写数据或接口章节。
- 幂等、内部接口设计、内部日志字段等技术实现问题，交给技术方案，不放 PRD 待确认项。
- 操作日志如果需要，只用一句话说明用途，不列内部字段。

---

## 9. 通知、权限、风控偏好

- 通知、权限、风控章节按需保留。
- 没有新增规则的小节直接删除。
- 通知模板、触发规则、渠道如已有公共模块，只引用公共模块；没有模板时不要假设存在。
- 风控限制如果未确认，只放待确认项，不写成开发范围。

---

## 10. 待确认项偏好

- 待确认项只保留真正影响产品范围、开发、测试、风控、上线验收的问题。
- 不会影响本期开发的问题，不放待确认项。
- 技术内部实现细节不放待确认项。
- 已经通过仓库资料查明结论的问题，不继续保留为待确认项。

---

## 11. Change Email PRD 经验教训

### 11.1 已沉淀偏好

- PRD 应面向开发测试，不面向“解释功能故事”。
- 复用 OTP 页面只写流转，不写场景参数和公共规则。
- 内部字段、内部接口、内部状态不写；涉及外部系统或跨模块边界时再写。
- 权限、风控、通知、验收测试不默认成章，按需保留。
- 待确认项只保留真正未定的产品 / 风控问题，不放技术实现问题。
- 成功后日志只需一句话，不写日志字段明细。
- 竞品和行业做法可用于建议，但未确认前不能写成开发范围。

### 11.2 避免再次出现的问题

- 不要把页面上能看出来的内容重复写成展示规则。
- 不要为了套模板把每个页面写成大表。
- 不要把公共页面规则复制到业务 PRD。
- 不要把内部技术设计写成产品待确认项。
- 不要把“本页不展示某入口”这类没有被要求的排除项写成边界，除非它是明确产品规则或安全规则。

---

## 12. 长期沉淀机制

每次 PRD 完成后，AI 需要输出简短经验教训总结：

1. 本次新增或强化的写作偏好。
2. 建议沉淀到本文的内容。
3. 是否需要更新标准 PRD 模板。

沉淀规则：

- 明确、稳定、可复用的 PRD 写作偏好，可以直接写入本文。
- 不确定是否长期适用的偏好，需要先询问用户。
- 单个业务需求结论只写入对应 PRD，不写入本文。
- 如果经验教训总结出需要调整 PRD 模板，AI 必须主动询问用户是否修改 `standard-prd-template.md`。
- 用户明确确认后，才允许修改 `standard-prd-template.md`。
- 模板结构变化写入 `standard-prd-template.md`；写作偏好和经验教训写入本文。
