# PRD Workflow v2.0：多 Agent 闸门式流程

> 本文档定义 PRD 写作、评审、确认、入库的标准流程。适用于新功能 PRD、迭代 PRD、历史 PRD 转译、PRD 修订与评审。

## 1. 核心原则

PRD 流程必须遵守以下原则：

1. **Canvas 优先**：新 PRD / 迭代 PRD 必须先生成 Brief 草稿到 Canvas，方便用户快速修改。
2. **用户确认闸门**：Brief 未经用户确认，不得生成正式 PRD 文件；用户未明确确认“更新到 Git”，不得修改仓库。
3. **多 Agent 分工**：写作、事实校验、模板校验、体验评审、技术评审、Git 写入必须职责隔离。
4. **Review 强制闸门**：PRD 写完后必须经过 Fact Review、Template Review、UX Review、Tech Review。
5. **Git 写入隔离**：Git Agent 只负责按用户已确认内容写入仓库，不得自行改需求、扩范围或新增文件。
6. **调研资料隔离**：调研资料不得塞入 Brief 主体，应独立存放，避免污染 PRD 读取。
7. **新增文件准入**：默认不新增一级目录；新增文件必须说明必要性、路径、生命周期和维护方式。
8. **长期经验反馈**：每次 PRD 完成后必须总结经验；如涉及模板或流程优化，必须询问用户确认后再修改。

## 2. 标准状态机

PRD 流程必须按以下状态推进：

```text
IDLE
  ↓
REQUEST_RECEIVED
  ↓
REQUIREMENT_CAPTURED
  ↓
BRIEF_DRAFTED_IN_CANVAS
  ↓
BRIEF_CONFIRMED_BY_USER
  ↓
FACT_CONTEXT_COLLECTED
  ↓
PRD_DRAFTED
  ↓
REVIEW_COMPLETED
  ↓
PRD_REVISED
  ↓
FINAL_QA_PASSED
  ↓
USER_APPROVED_FOR_GIT
  ↓
GIT_UPDATED
  ↓
EXPERIENCE_RECORDED
  ↓
DONE
```

禁止跳过以下闸门：

```text
BRIEF_DRAFTED_IN_CANVAS → GIT_UPDATED
PRD_DRAFTED → GIT_UPDATED
REVIEW_COMPLETED → GIT_UPDATED
```

必须经过：

```text
USER_APPROVED_FOR_GIT
```

## 3. 标准流程

### 3.1 需求接收

当用户发起 PRD 需求时，先由 Task Router 判断是否进入 PRD Workflow。

进入条件包括但不限于：

- 用户明确说“PRD 流程”
- 用户要求写功能需求
- 用户要求基于历史 PRD 转译
- 用户要求修订某个 PRD
- 用户要求对 PRD 做评审并修正

不进入条件：

- 普通问答
- 单纯解释概念
- 只要求查看文件，不要求产出 PRD

### 3.2 需求澄清

Requirement Agent 负责整理用户输入，输出：

- 需求背景
- 目标用户
- 用户问题
- 业务目标
- 范围边界
- 已知约束
- 待确认项

如缺少关键信息，优先在 Canvas Brief 中以 TBD 形式标记，不得自行编造。

### 3.3 Brief 草稿

Brief Agent 必须先生成 Canvas Brief。

Brief 只包含：

- 需求背景
- 用户问题
- 目标
- 范围
- 关键流程
- 待确认项
- 明确不做什么
- 初步页面 / 系统边界

Brief 不应包含大量调研过程、历史资料堆叠、未验证事实或完整 PRD 正文。

### 3.4 用户确认 Brief

用户确认 Brief 后，才允许进入正式 PRD 写作。

用户未确认前，禁止：

- 新增 PRD 文件
- 修改 Git 仓库
- 提交正式 PRD
- 创建正式评审结论文件

### 3.5 事实上下文收集

Fact Agent 负责读取历史 PRD、配置表、事实依据和用户指定附件。

Fact Agent 输出必须包含：

```json
{
  "status": "pass | fail | uncertain",
  "facts_used": [],
  "missing_evidence": [],
  "risk_points": []
}
```

规则：

- 有依据的内容才能写成事实。
- 无依据但合理的内容只能写成假设、待确认项或建议。
- 不得把推测写成系统事实。

### 3.6 PRD 草稿

PM Agent 根据确认后的 Brief、事实上下文和 PRD 模板生成正式 PRD 草稿。

PM Agent 可以写：

- 需求背景
- 目标与范围
- 用户故事
- 业务流程
- 页面说明
- 状态与异常
- 权限与风控
- 数据与埋点
- 待确认项

PM Agent 不得：

- 未经用户确认直接更新 Git
- 自行扩大需求范围
- 自行新增一级目录
- 把无依据内容写成事实

### 3.7 强制评审

PRD 草稿完成后，必须经过以下评审：

#### 3.7.1 Fact Review

检查：

- 是否有事实依据
- 是否引用了错误历史资料
- 是否编造系统能力
- 是否遗漏用户已确认内容
- 是否把假设写成事实

Fact Review 不通过，禁止提交 Git。

#### 3.7.2 Template Review

检查：

- 是否符合标准 PRD 模板
- 章节是否完整
- 标题层级是否一致
- 待确认项是否规范
- 业务流程是否清楚

Template Review 不通过，禁止标记 PRD 完成。

#### 3.7.3 UX Review

检查：

- 用户路径是否闭环
- 入口、返回、失败、重试是否清楚
- 空状态、异常状态、错误提示是否完整
- 关键操作是否有反馈
- 是否存在 P0 / P1 用户体验问题

UX Review 存在 P0 / P1 问题，必须修正后再进入最终确认。

#### 3.7.4 Tech Review

检查：

- 系统交互是否清楚
- 接口依赖是否明确
- 状态机是否合理
- 兼容性、幂等性、安全边界是否说明
- 是否存在明显不可落地问题

Tech Review 发现阻断问题时，必须回到 PM Agent 修正。

### 3.8 PRD 修正

PM Agent 根据评审意见修正 PRD。

修正要求：

- 必须逐条响应评审意见。
- 不得选择性忽略 P0 / P1 问题。
- 如果评审意见互相冲突，由 Orchestrator 收敛方案并请用户确认。

### 3.9 最终 QA

QA Agent 负责最终一致性核验：

- PRD 是否覆盖用户确认的 Brief
- 是否存在未处理评审意见
- 是否存在事实风险
- 是否符合模板
- 是否准备好进入 Git

QA 通过后，进入用户确认阶段。

### 3.10 用户确认 Git 写入

必须由用户明确确认，才允许 Git Agent 执行写入。

有效确认包括：

- “更新到 Git”
- “提交到仓库”
- “写进去”
- “同意，执行”
- 其他语义明确的授权表达

如果用户只是说“看看”“评审一下”“给方案”，不得写 Git。

### 3.11 Git 写入

Git Agent 只负责按已确认内容写入仓库。

Git Agent 不得：

- 自行修改需求
- 自行新增一级目录
- 自行扩大修订范围
- 未说明原因删除文件
- 未经用户确认修改模板

### 3.12 经验总结

Experience Agent 在流程结束后输出：

1. 本次用户偏好
2. 本次 PRD 模板暴露的问题
3. 是否建议修改 PRD 模板
4. 是否建议修改 Workflow
5. 是否建议写入长期记忆

如涉及模板或流程优化，必须询问用户确认：

```text
本次经验暴露出模板问题：xxx。是否更新 PRD 模板？
```

用户确认后，才允许修改模板或流程。

## 4. Agent 职责与权限

### 4.1 Orchestrator Agent

职责：

- 判断当前状态
- 决定调用哪个 Agent
- 检查是否满足进入下一阶段
- 阻止越权操作

权限限制：

- 不直接写正式 PRD 正文
- 不直接提交 Git

### 4.2 Requirement Agent

职责：

- 整理用户需求
- 标记缺失信息
- 输出 Brief 输入

权限限制：

- 不直接写 PRD
- 不直接更新 Git

### 4.3 Brief Agent

职责：

- 生成 Canvas Brief
- 帮助用户快速修改需求草稿

权限限制：

- 不生成正式 PRD 文件
- 不更新 Git

### 4.4 Fact Agent

职责：

- 查历史 PRD、配置、事实资料
- 判断是否有依据
- 输出事实风险

权限限制：

- 不改 PRD
- 不做产品决策
- 不把推测写成事实

### 4.5 PM Agent

职责：

- 写 PRD 草稿
- 根据评审意见修正 PRD

权限限制：

- 不直接提交 Git
- 不自行扩大范围

### 4.6 UX Agent

职责：

- 检查用户路径
- 检查异常状态
- 检查提示语、空状态、失败状态、重复提交、返回路径

权限限制：

- 只评审，不直接修改 Git

### 4.7 Template Agent

职责：

- 检查 PRD 是否符合标准模板
- 检查章节结构、标题层级、待确认项格式

权限限制：

- 不判断业务优劣
- 不直接修改 Git

### 4.8 Tech Agent

职责：

- 检查接口、状态、系统交互、兼容性、数据依赖

权限限制：

- 不替 PM 做产品决策
- 不直接更新 Git

### 4.9 Git Agent

职责：

- 根据用户已确认内容更新仓库
- 保持文件路径和修改范围可解释

权限限制：

- 不自行改需求
- 不自行新增文件
- 不自行修改模板

### 4.10 QA Agent

职责：

- 最终一致性检查
- 检查 Git 内容是否和用户确认一致

权限限制：

- 只核验，不扩范围

## 5. 调研资料管理

调研资料应放入：

```text
references/research-notes/
```

调研资料用于记录：

- 外部资料链接
- 历史 PRD 依据
- 配置表依据
- 竞品观察
- 待验证事实
- 调研结论摘要

禁止：

- 把调研过程塞入 Brief 主体
- 把未经验证的调研内容直接写成 PRD 事实
- 让 PRD 自动读取大量原始调研材料

PRD 只能引用经过 Fact Agent 摘要后的事实结论。

## 6. 新增文件准入规则

默认不新增一级目录。
默认不新增流程文件。
默认优先修改既有文档。

只有满足以下任一条件，才允许建议新增文件：

1. 新内容生命周期明显独立。
2. 放入现有文件会造成模板污染。
3. 会被多个 PRD 复用。
4. 属于长期配置、事实库或规范。
5. 用户明确要求新增。

每次建议新增文件时，必须说明：

- 为什么不能放进现有文件
- 建议路径
- 生命周期
- 后续维护方式
- 是否会影响 PRD 读取

## 7. 评审结论格式

所有评审 Agent 输出必须使用以下结论：

```text
通过 / 条件通过 / 不通过
```

评审意见必须区分优先级：

```text
P0：阻断问题，必须修正
P1：高优先级问题，原则上必须修正
P2：优化建议，可择机修正
P3：非必要建议
```

## 8. 用户确认规则

以下表达视为允许进入下一阶段，但不等于允许 Git 写入：

- “继续”
- “可以”
- “按这个来”
- “同意这个方案”

以下表达才视为允许 Git 写入：

- “更新到 Git”
- “提交到仓库”
- “写进去”
- “同意，执行”
- “改吧”
- 其他语义明确的写入授权

如果上下文存在歧义，Orchestrator 必须优先询问确认，不得擅自写入。

## 9. 最小执行版本

第一阶段至少实现以下链路：

```text
用户需求
  ↓
Brief Canvas
  ↓
用户确认
  ↓
PRD 草稿
  ↓
Fact Review + Template Review + UX Review
  ↓
PM 修正
  ↓
用户确认
  ↓
Git 更新
  ↓
经验总结
```

Tech Review、Research Notes、Replay Harness 可在后续增强，但不应破坏上述核心闸门。
