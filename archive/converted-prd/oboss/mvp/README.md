**2.1.1 Popup管理**

Popup功能管理的是应用内的运营目的弹窗，当前主要作用是引导用户进入活动页面。App端对多种弹窗类型的展示和优先级规则，由业务侧进行规范。

页面入口：Operation\>Popups；

在Popups系统内的Business line和Display Location配置项，AIX需要与Atome进行隔离

Business line：本期仅「AIX」

Display Location：本期

「AIX/Homepage」，对应AIX应用的首页

Popups管理列表

管理列表分Acitve和Inactive两个tab，分别筛选展示生效和失效状态的弹窗配置

生效条件：配置的Status=ture **且** 配置的结束展示时间晚于当前时间

**4.1.5 User Group使用场景**

用户组（User Group）核心支撑两类业务场景，覆盖 “实时校验” 与 “提前圈选” 两大核心需求，适配不同业务运营与功能配置场景：

实时校验

场景定义：基于用户组配置的多维度条件组合，实时校验单个用户是否符合规则，进而触发差异化的功能展示或交互逻辑。

核心逻辑：以用户唯一标识（UID）为检索依据，实时查询该用户是否满足用户组的配置条件（如属性、标签、行为等组合规则）。

典型应用：主要用于与 APP 端、campaign/task等功能的发放判断等场景

人群圈选

场景定义：基于预设的配置条件（如用户属性、行为标签、业务状态等），在指定时间筛选出符合条件的用户集合（用户 UID 列表），用于定向运营动作。

核心逻辑：系统根据用户组规则，在指定时间计算并存储满足条件的全量用户列表，无需实时校验，可直接调用该用户集合。

典型应用：主要用于营销触达、定向发券、定向发放任务等场景

**6 System Users**

**6.1 权限系统概述**

权限体系核心逻辑

系统采用 “权限点 - 权限角色 - 用户” 三层关联逻辑，实现页面展示与功能使用的精细化权限管控，确保用户仅能访问与操作具备权限的功能模块。

权限点：

权限点由研发团队在开发页面或功能时按需设定，作为系统功能的最小权限单元，明确标识某一页面的访问权限或某一功能的操作权限。

命名规范：采用 “模块 / 功能 - 操作类型” 格式命名，确保权限点含义清晰、可追溯。

权限角色

权限角色是权限点的集合，用于关联一组业务相关的权限点

每个权限角色可灵活配置关联的权限点，支持根据业务角色的职责范围，组合不同的权限点形成专属规则。

用户权限

关联方式：通过赋予用户权限角色，建立用户与权限点的间接关联关系（即用户获得某一权限角色后，自动拥有该规则下所有权限点的权限）。

权限校验逻辑：当用户使用某一功能（如访问页面、执行操作）时，系统会实时校验用户是否拥有该功能对应的权限点；若拥有则允许使用，否则拦截操作并提示权限不足。

用户与lark相关联，仅有我司lark账号可以申请权限以及左右权限的审批人。当用户失去lark身份，则对应失去系统的用户身份。

Expiry date required：必填开关组件，用于设置该角色是否需要过期日期：

开启（选中）：需配置角色的过期日期，适用于临时权限场景；

关闭（未选中）：无需配置过期日期，适用于长期权限场景。

检验规则

必填校验：提交前需确保已填写或配置。

审批人规则：Approver/s至少选择 1 位审批人，否则无法提交，保障权限变更的审批合规性。

权限点管理：可根据业务需求新增或删除Authorities中的权限点，实现角色权限的动态调整；权限点由研发预设，编辑时仅支持选择已有权限点。

过期日期逻辑：若Expiry date required开启，需在后续流程中配置具体过期日期。

权限审批

权限审批与Lark的审批系统进行关联

消息提醒：审批人在 Lark 端收到消息通知（包含审批单摘要：申请人、申请角色、申请理由），点击通知可进入审批详情页。

审批操作：Lark 审批详情页展示完整申请信息（与系统提交内容一致），支持两种操作：

通过：审批人点击 “通过” 按钮，Lark 系统记录审批结果并同步时间，并同步至权限管理系统。

权限管理系统接收 Lark “通过” 结果后，自动执行权限赋予：自动为申请用户赋予所申请的权限角色，关联用户与角色的关联关系。

系统向申请用户推送 Lark 通知，告知用户申请成功。

拒绝：审批人点击 “拒绝” 按钮，需填写拒绝原因（必填），Lark 系统记录结果后同步至权限管理系统。

权限管理系统接收 Lark “拒绝” 结果后，不执行权限赋予操作，仅记录审批状态为 “已拒绝”。

系统向申请用户推送 Lark 通知，告知用户申请驳回。
